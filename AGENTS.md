## Product And Architecture Rules

- Highlander Today is a multi-tenant local community platform. Preserve tenant-aware behavior; do not hardcode single-tenant assumptions into new shared code.
- Prefer the existing Next.js + TypeScript + Prisma architecture over adding sidecar services unless a plan explicitly calls for it.
- Use the existing auth, trust, role, and moderation model. Do not bypass those boundaries for convenience.
- Keep public AI/reporting flows constrained, source-grounded, reviewable, and human-approved before publication.
- Preserve the product’s “local digital infrastructure” posture. Favor accountable workflows over generic platform patterns.

## Codebase Rules

- Canonical Prisma schema path is `prisma/schema.prisma`. Do not create or rely on a root-level `schema.prisma`.
- Database access should use the shared `db` path already established in the repo.
- Active auth configuration lives at `src/app/api/auth/[...nextauth]/route.ts`.
- Do not reintroduce a Prisma session adapter for auth; this project relies on JWT-session behavior.
- Middleware-provided request headers for `/api/*` are part of the active contract. Do not silently remove or bypass that assumption in route work.
- The canonical article route family is `/local-life/*`. Legacy `/articles/*` behavior is compatibility-only.
- Categories and navigation are database-driven. Do not hardcode category names, slugs, or menu structure when dynamic data already exists.

## UI And Design Rules

- Reuse the shared structural vocabulary from [DESIGN-SYSTEM-ARCHITECTURE.md](./DESIGN-SYSTEM-ARCHITECTURE.md): `page-header`, `section-*`, `card-*`, and related semantic classes.
- Prefer shared semantic theme classes and tokens over route-local hardcoded colors.
- For admin record-management surfaces, default to the compact `admin-list` table pattern documented in [ADMIN-LIST-DESIGN.md](./ADMIN-LIST-DESIGN.md).
- When one admin area mixes management and creation, prefer nested secondary tabs such as `List` and `+ Form` rather than one long mixed panel.
- Keep mobile behavior and accessibility in scope. Shared shell/header/dialog changes should preserve keyboard, focus, and screen-reader behavior.

## Working Style

- Inspect existing patterns in nearby files before adding abstractions.
- Keep edits scoped to the task. Do not opportunistically refactor unrelated areas.
- Do not overwrite or revert user changes in a dirty worktree unless explicitly asked.
- Prefer incremental improvements over broad rewrites.
- Add tests in proportion to risk, especially for shared behavior, route logic, Prisma-backed flows, permissions, and reporter workflows.

## Verification Expectations

For code changes, run the narrowest meaningful verification first, then broader checks as needed.

Common commands:

- `npm run typecheck`
- `npm run lint`
- `npm run test:unit`

Reporter, route, and service changes should usually include targeted unit coverage when practical.

## Database And Local Development Notes

- Local Postgres runs on port `5433`, not `5432`.
- Theme manifests are generated by `scripts/generate-theme-manifests.js` via package pre-scripts. If a new manifest file is added under `src/lib/theme/manifests/`, restart the relevant dev/build command rather than hand-editing the generated registry.
- Schema-changing work may require `prisma generate` and `prisma db push` against the intended environment before the app behaves correctly.
